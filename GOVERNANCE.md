# Governance

FlappyLearn uses lightweight maintainer governance. The goal is to move quickly while keeping the project reliable, educational, and welcoming.

## Project Roles

**Maintainers** are responsible for releases, roadmap direction, security handling, moderation, and final merge decisions.

**Core contributors** regularly improve code, docs, tests, examples, or community support and may be invited to review pull requests in their areas.

**Contributors** submit issues, discussions, pull requests, experiments, benchmarks, docs, and educational materials.

**Users and learners** run experiments, ask questions, share results, and help reveal where the project is confusing.

## Decision Process

Most decisions happen by lazy consensus:

1. A contributor opens an issue, discussion, or pull request.
2. Maintainers and contributors discuss tradeoffs.
3. If there is no serious objection, the change proceeds.
4. For major architecture, governance, release, or compatibility decisions, maintainers make the final call after public discussion.

Maintainers should explain rejected proposals clearly and kindly.

## Maintainer Responsibilities

- Keep CI passing and releases reproducible.
- Protect deterministic evaluation semantics.
- Prioritize beginner-friendly docs and onboarding.
- Review security reports promptly.
- Moderate community spaces according to the code of conduct.
- Avoid merging major public behavior changes without tests and docs.

## Contributor Path

Contributors build trust by:

- Opening focused pull requests.
- Responding constructively to review.
- Improving tests and docs.
- Helping answer community questions.
- Preserving project values around clarity and reproducibility.

## Project Values

- Learning should be visible.
- Experiments should be reproducible.
- Small models can teach big ideas.
- Documentation is part of the product.
- Community health is technical infrastructure.
